# AI事前問診メモ ─ クライアント医院への提供に向けた検討事項

作成日: 2026/06/21
対象: 本アプリ（AI事前問診メモ）を、デモ段階から実際のクライアント医院へ提供するための検討メモ

---

## 0. 位置づけ（デモと本番は別物）

- **2026/7/2 のデモ**: Cloudflare Pages 配信で問題なし。速い・無料・HTTPS・個人名なしURL（`qureas-monshin.pages.dev`）。実患者のデータを継続的に蓄積しない前提なのでOK。
- **クライアント医院への提供（本番）**: 患者の症状＝**健康情報**を扱うため、法令・ガイドライン・契約・運用の要件が一気に増える。以下はそのための検討事項。

---

## 1. 一番の論点：医療情報は「要配慮個人情報」

- 患者の症状は個人情報保護法上の **要配慮個人情報**。取得には本人同意・安全管理措置が必要。
- 医療機関向けには **3省2ガイドライン** への準拠が問われる:
  - 厚労省「医療情報システムの安全管理に関するガイドライン」
  - 総務省・経産省「医療情報を取り扱う情報システム・サービスの提供事業者における安全管理ガイドライン」
- クリニックは医療機関として責任を負う。**提供側（自社）が「準拠できる仕組み」を用意**する必要がある。

---

## 2. 検討すべき4つの柱

### ① データの送り先（最大の課題：OpenAI）
- 現状: 患者の発話（健康情報）を **OpenAI（米国）** に送信している。
- 本番では「越境移転」「国内保管」の観点で見直しが必要。
- 現実的な解: **Azure OpenAI Service（東京リージョン）** へ切り替え
  - データを日本国内で処理／学習に使われない／企業向け契約（DPA）あり
  - コード変更は比較的小さい（送り先URL・キー・リクエスト形式の微調整）
- 代替: OpenAI Enterprise/API（zero-retention・no-training＋DPA）。ただし処理は米国のまま。
- → **本番化の最初の一手はここ**（コンプラ上のインパクト最大）。

### ② ホスティング／データ保管
- 本番は国内リージョン・ガイドライン対応のインフラを検討（AWS東京を適切に構成、さくら/ニフクラ等の医療グレード）。
- **現設計の強み**: 問診メモは **QRで端末→受付に渡し、サーバーに患者データを溜めていない**。
  - 「サーバーに健康情報を保存しない」設計を維持できれば、リスクもガイドライン適用範囲も大きく下げられる。
  - → **この方針は本番でも死守する価値あり**。

### ③ 仕組み（複数医院対応・マルチテナント）
- 問診テンプレートは `clinic-template.js` で分離済み＝マルチテナント化しやすい設計。
- 追加で必要: 医院ごとの識別子/サブドメイン・ブランディング、受付スタッフのログイン認証、アクセス制御、監査ログ。

### ④ 契約・運用・コスト
- 患者への**同意取得フロー**（AIに送る旨・目的の明示）、利用規約・プライバシーポリシー。
- **データ処理契約**（医院との間／AI事業者との間）。
- API課金の管理、医院への**課金モデル**、サポート・SLA・障害対応・バックアップ。
- セキュリティ: 脆弱性診断、レート制限、WAF。

## 4. 次の一歩のおすすめ

1. **7/2 はこのまま実施**（デモとして十分）。
2. パイロットに進むなら、**まず「AI送信先を Azure OpenAI 東京」へ**（インパクト最大・変更は比較的小）。
3. **「サーバーに患者データを保存しない」現設計を維持**（強み）。

---

## 5. 重要な注意

- 医療情報・3省2ガイドライン・個人情報の越境移転は**専門領域**。
- 本メモは一般的な整理であり、**法的助言ではない**。
- 本番提供の前に、**医療情報に詳しい弁護士・セキュリティ専門家**と、**対象医院のセキュリティ方針**を必ず確認すること。
- 参考: 手元資料（厚労省ガイドライン、医療情報を取り扱う情報システムのガイドライン経産省、医療機関用サイバーセキュリティチェックリスト 等）。
